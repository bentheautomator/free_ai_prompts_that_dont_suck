# Contributing

## Adding a New Prompt

### 1. Pick a category

Put your prompt in the right `prompts/<category>/` directory:

| Category | What Goes Here |
|----------|---------------|
| `code-safety` | Preventing data loss, destructive actions, unauthorized deletion |
| `code-quality` | Correct edits, following patterns, reading before writing |
| `instruction-following` | Making AI follow user-defined rules and processes |
| `git` | Safe git operations, commit messages, branch protection |
| `communication` | AI tells you what it's doing, asks before acting |
| `context` | Preventing hallucinated assumptions, verifying before acting |
| `scope` | Keeping changes focused, preventing over-engineering |

Need a new category? Just create the directory. The build script discovers categories automatically.

### 2. Use the template

Every prompt file needs YAML frontmatter and follows this structure:

```markdown
---
title: Your Prompt Title
slug: your-prompt-title
category: code-safety
tags: [universal]
works_with: all
severity: high
one_liner: "Short description of what it prevents"
---

# Your Prompt Title

> **Works with:** All AI coding assistants (Claude Code, Cursor, Windsurf, Copilot, Cline, Aider)

> One-line description of what it prevents.

## The Problem

What goes wrong without this instruction. Be specific — name the failure mode,
describe what happens, explain why AI assistants do this by default.

## The Instruction

Copy everything between the horizontal rules into your instructions file:

---

### Your Prompt Title

THE ACTUAL INSTRUCTION BLOCK GOES HERE.
This is what gets extracted into install/ files.
Make it copy-paste ready — no references to other sections.

---

## Why It Works

What makes this instruction effective. Name the mechanism — don't just say
"it helps." Explain what cognitive or behavioral pattern it exploits.

## Origin

Real incident that created this rule. Optional but strongly encouraged.
The more specific, the more credible.
```

### 3. Frontmatter fields

| Field | Required | Notes |
|-------|----------|-------|
| `title` | Yes | Human-readable name |
| `slug` | Yes | Must match filename (without .md) |
| `category` | Yes | Must match parent directory name |
| `tags` | Yes | Array. Add `essential` for starter pack inclusion |
| `works_with` | Yes | `all` or list like `[claude-code, cursor]` |
| `severity` | Yes | `critical`, `high`, or `medium` |
| `one_liner` | Yes | Under 80 chars. Used in the README table. |

### 4. Build and verify

```bash
make build      # Regenerates install/ files and README table
make validate   # Just validates, no file generation
```

The build script will catch:
- Missing frontmatter fields
- Slug/filename mismatches
- Category/directory mismatches
- Missing instruction blocks

### 5. Open a PR

Your PR should include:
- The new prompt file in `prompts/<category>/`
- Updated `install/` files (generated by `make build`)
- Updated README table (generated by `make build`)

## What We're Looking For

- **Real failure modes.** Every prompt must trace back to something that actually went wrong.
- **Specific instructions.** Not "be careful" but "never do X without doing Y first."
- **Red flags section.** Name the thought patterns that precede the failure.
- **Universal applicability.** Prompts should work across AI coding assistants, not just one tool.

## What We Don't Want

- Vibes. "Be a good coding assistant" is not a prompt.
- Tool-specific features dressed up as instructions. If it only works in one tool, note that clearly.
- Duplicates. Check existing prompts before adding. If yours overlaps, consider strengthening the existing one.
- Prompt injection or jailbreaks. Obviously.

## Style

- Direct. No hedging.
- Written for practitioners, not beginners.
- If it's true and it's funny, say it.
- Evidence over vibes.
